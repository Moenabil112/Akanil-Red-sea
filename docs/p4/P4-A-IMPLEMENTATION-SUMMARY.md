# P4-A — Implementation Summary

Branch: `feature/p4a-internal-secure-case-management` · Base: `main` (merged
P0 `3b2b4c6` + P1 `101ed96` + P2 `de94df5` + P3 `a2a2293`). **Merged into
`main` via PR #7 (merge commit `eb3f02c`)** and accepted as the P4-B baseline;
not deployed, no production infrastructure configured.

P4-A adds an **internal, employee-only** secure case-management foundation
alongside the unchanged public Gateway. The first release is restricted to
administratively provisioned Akanil employees: **no external accounts, no
self-registration, no client/participant portal, no data room, no file upload,
no external notification, no calendar, no automatic matching, no AI
decision-making, no public case tracking.**

## Architecture

- **Next.js App Router** (existing) + **PostgreSQL 16** + **Prisma 6.19** +
  **Argon2id** (`@node-rs/argon2`) + **Zod**. Custom database-backed sessions
  (Auth.js Credentials cannot use DB sessions — see ADR-021).
- Internal app under `/[lang]/internal/*` (ar/fr/en, Arabic native RTL),
  separated from the public site by robots, sitemap exclusion, noindex/no-store
  middleware headers, a strict CSP, the `P4_INTERNAL_ENABLED` flag (default
  false, fail-closed) and server-side authorization on every route and mutation.
- The public Digital Reception Lite is unchanged and stays disconnected: an
  employee manually creates a case from the institutional inbox. The public
  build compiles with no database.

## Data model (15 entities)

Employee · Session · Organization · Contact · Case · CaseAssignment ·
InternalNote · QualificationReview · InformationGap · EvidenceReference ·
MeetingPreparation · MeetingRecord · Decision · Commitment · AuditEvent
(+ a transaction-safe Counter for case references). Public controlled
taxonomies (request types, platform / value-chain / Forum ids) are validated
strings, never redefined. See `P4-A-DATA-MODEL-AND-CASE-LIFECYCLE.md`.

## Security and access

- Argon2id hashing, ≥14-char passwords, generic auth errors, HttpOnly/SameSite/
  Secure cookies, absolute (8h) + idle (60m) session expiry, lockout (5 fails /
  15 min), forced first-login change, session revocation on reset/disable.
- Six roles (SYSTEM_ADMIN, OPERATIONS_MANAGER, CASE_MANAGER, SPECIALIST_REVIEWER,
  FORUM_COORDINATOR, READ_ONLY_AUDITOR) driven by a single default-deny
  permission matrix, plus object-level case access. See `P4-A-RBAC-MATRIX.md`
  and `P4-A-SECURITY-AND-ACCESS-BASELINE.md`.

## Case lifecycle

A server-enforced ten-state machine (NEW → … → CLOSED) with explicit
transitions, mandatory closure reason, role-restricted reopening, optimistic
concurrency and an audit event per transition. The internal case reference
(`AKN-YYYY-000001`) is generated by a transaction-safe counter, is immutable,
and is never shown publicly.

## Interface

Login · forced change-password · dashboard · cases list (server-side filter and
safe search) · new case (manual, with authorized-channel confirmation and a
sensitive-data warning) · case detail (overview, context, qualification,
assignments, notes, gaps, evidence, decisions, commitments, audit summary) ·
organizations list/detail · My-work queue · audit trail · user administration.
Restrained operations styling — not a public dashboard, CRM or marketplace.

## Provisioning and operations

Admin CLI: `internal:user:create` / `:disable` / `:reset-password` (passwords
read from the environment, never argv; hashes/passwords never printed).
Database: Docker Compose PostgreSQL, `db:generate` / `db:migrate` /
`db:migrate:test` / `db:seed:roles` / `db:studio`; migrations committed and
never auto-run at startup. Local backup/restore smoke script. See
`P4-A-LOCAL-OPERATIONS-RUNBOOK.md`.

## Tests, build and CI

- `npm run typecheck` — 0 errors.
- `npm test` — unit + component (adds RBAC/lifecycle/reference/password and
  boundary/prohibited-feature/privacy scans).
- `npm run test:integration` — PostgreSQL integration (transactional case
  creation, unique references, valid/invalid transitions, concurrency conflict,
  immutable reference, decision immutability, single-owner history, archive
  preserves cases).
- `npm run build` — public build succeeds **without a database**.
- `npm run lint` — clean.
- CI adds a PostgreSQL service, `db:generate`, `db:migrate:test` and
  `test:integration`; synthetic CI credentials only, no repo secret contexts,
  no deploy.

## Scope confirmation

No production deployment occurred; no cloud database or hosted identity service
was connected. The internal system defaults to disabled. The public reception
remains disconnected from the internal database. P4-B and later operational
features (notifications, scheduling, uploads/data room, AI assistance, external
accounts) are recorded as future work and were not implemented.
